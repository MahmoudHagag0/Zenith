# M4-003.2_DASHBOARD_HIGH_FIDELITY_SPECIFICATION

**Document ID:** ZOS-M4-003.2
**Version:** 1.0.0
**Status:** Proposed — Awaiting Product Leadership Review
**Owner:** Architecture Team (leadership-authored philosophy; engineering-maintained specification)
**Milestone:** M4 — Design Foundation (Dashboard Design phase)

------------------------------------------------------------------------

# Purpose

Assigns Dashboard's final wireframe (`M4-003.1` §3) its concrete
D2-series tokens and component mappings — typography, spacing,
elevation, color, chart rules, states, interaction, motion,
accessibility. **No new token, color, or component is introduced here.**
Every value below already exists in D2-001 through D2-007; this
document only selects and justifies which existing value applies to
which Dashboard block. Specification only — no implementation code.

------------------------------------------------------------------------

# 1. Color & Human Behavior Review (Performed Before Assignment)

Reviewed against cognitive psychology, human perception, behavioral
science, HCI, accessibility, long-session visual comfort, contrast
perception, fatigue reduction, and decision support — per this task's
own required lenses, and explicitly **not** against unevidenced
"color psychology" claims (Constitution §8.4, D1-001 §4 already
disconfirm those as a design basis).

## 1.1 Does Dashboard need any new color?

**Reviewed and rejected.** The most common temptation at this stage —
give the Decision Readiness Summary (`DASH-002`) a distinctive "hero"
color treatment (a colored banner, a gradient background, an
accent-tinted panel) to make it feel important — is explicitly rejected.
D1-003 §3 reserves the accent hue exclusively for "you can act here";
using it as a background wash on `DASH-002` would teach the trader a
false association (accent = "this is the important reading") that
contradicts its own defined meaning everywhere else in the product
(navigation, buttons, focus rings), violating Design Constitution rule
1 (one color vocabulary, consistently applied). `DASH-002`'s own
importance is communicated by **position, size, and typography weight**
(§8.1 below) — exactly per Constitution §8.1/§14 rule 6 ("visual weight,"
not hue, marks the most decision-relevant element).

## 1.2 Long-session comfort (6–10 hour sessions)

**Reviewed.** `surface.base` (D2-002 §5) is deliberately not pure white
— the documented, already-corrected long-session finding from D2-002's
own review (§5, "a pure-white base is a known contributor to
glare/eye strain across a sustained 6–10 hour viewing session"). Since
Dashboard is the single most-revisited screen in the product (session
start and every return, `M4-002` §2), it is the screen where this
correction matters most in practice. **No further change required here
— confirming the existing correction is being inherited correctly, not
introducing a new one.**

## 1.3 Fatigue reduction via signal discipline

**Reviewed.** `DASH-002`/`DASH-006`'s own evidentiary reads never use
`signal.critical`/`signal.warn` for ordinary favorable/unfavorable
direction (D1-003 §4, "reserved exclusively for genuinely disclosed,
evidence-backed [operational] conditions"). Gain/loss uses
`data.positive`/`data.negative` only (D2-002 §11), at markedly lower
saturation than the signal family — meaning a trader scanning
Dashboard repeatedly across a session never encounters a false-alarm
read merely because a position is down. **Confirmed correct.**

## 1.4 Decision quality / no manufactured emotion

**Reviewed.** No block introduces motion-on-update beyond
`motion.easing.standard` (D2-001 §7) — no flash, pulse, or color-shake
on a reading changing (D1-002 §5.3). No countdown, badge, or
unread-style indicator appears anywhere on Dashboard (D1-002 §6.2) —
confirmed against the final wireframe (`M4-003.1` §3): the consolidated
Peripheral row (§2.6 of that document) uses plain text labels, not
notification-style badges, precisely to avoid implying an unread count
that does not exist.

## 1.5 Accessibility / contrast perception

**Reviewed.** Every text/background pairing used on Dashboard
(`text.primary`/`text.secondary`/`text.muted` on `surface.base`/
`surface.raised`) is already independently WCAG 2.1 AA-verified at the
system level (D2-002 §8) — Dashboard introduces no new pairing requiring
separate verification.

**Conclusion of Color Review:** Dashboard uses the existing D2-002
palette exactly as defined, introduces zero new colors, and rejects the
one new-color temptation identified (§1.1). Every color decision below
is therefore a *selection*, not a *creation*.

# 2. Typography Assignment

Per D2-003 §1, applied to the final wireframe (`M4-003.1` §3):

| Block | Element | Token |
|---|---|---|
| `DASH-002` | Synthesized conclusion statement | `text.heading` (28px/600) — the single largest text on the page, per D1-002 §1.3 (visual weight by Attention level) |
| `DASH-002` | Supporting reasoning line(s) | `text.body` (16px/400) |
| Nested `DASH-003` | Confidence statement + uncertainty statement | `text.body.small`/`text.caption` (14px), identical size/weight for both (Design Constitution rule 8, D1-002 §7.1 — confidence and uncertainty never differ in visual prominence) |
| `DASH-004`/`DASH-005`/`DASH-006` | Card title | `text.title` (22px/600) |
| `DASH-004`/`DASH-005`/`DASH-006` | Preview/summary line | `text.body.small` (14px) |
| Any numeric figure (P/L, price, confidence %) anywhere | — | `text.numeric`, tabular figures mandatory (D2-003 §5.1) — never `text.body` for a number compared against another number |
| Peripheral row labels | Area name | `text.micro` (12px), `letterSpacing.wide` if rendered uppercase |
| Timestamps/metadata (e.g. "as of 9:31 AM") | — | `text.micro` |

Rule 3.1 (D1-002): no font size outside this table is used anywhere on
Dashboard.

Per D2-001 §5 and D2-005:

| Block | Component family (D2-005) | Elevation | Radius |
|---|---|---|---|
| `DASH-002` region | Panel (§3) — the screen's own primary content area, holding sole Primary Attention (D2-005 §3 constraint) | `elevation.0` (a panel is the page, not raised above it) | `radius.lg` if a bounding edge is used at all |
| Nested `DASH-003` | No separate component — rendered as sub-content inside the Panel, not a Card (per `M4-003.1` §2.2's fix; giving it its own Card/elevation would recreate the sibling-region problem already fixed) | Inherits parent | Inherits parent |
| `DASH-006`, `DASH-005`, `DASH-004` | Card (D2-005 §2) | `elevation.1` (raised surface on the page background) | `radius.md` |

| Loading state (any block) | Skeleton (D2-005 §17) | Same elevation as the eventual Success state — never a distinct "loading chrome" | Same as parent |

elevation or fill color to imply favorability — a losing-position
Portfolio Snapshot and a gaining-position one use an identical Card
treatment (Constitution §7.3, §12.4).

# 6. Chart Rules

Dashboard V1 contains no chart component — `DASH-002`, `DASH-005`, and
`DASH-006` are synthesized-text and tabular-figure summaries, not
visualizations, per `26` §3's own component definitions. This is stated
explicitly so no future implementation adds one without a separate
decision: if a future Dashboard revision introduces a sparkline or
trend indicator on a KPI-style element, it must satisfy D2-006 §6.3
("no sparkline or trend chart... competes with the primary value for
visual weight... rendered small and low-contrast, confirming the
number, not competing with it") — stated here as a forward constraint,
not an authorization to add one now.

# 7. States

Per `26` §3 (per-component) and D2-005:

| State | Treatment |
|---|---|
| Loading | Skeleton (D2-005 §17), identical footprint to Success state, slow/low-contrast pulse, `motion.duration.slow`-paced |
| Empty (`DASH-002`: "no clear opportunity") | Rendered with identical calm tone/weight as Success — never smaller, grayed, or apologetic (D1-002 §14.3, Constitution §12.4) |
| Empty (`DASH-005`: no Watchlist entries) | Factual, states why and what a legitimate next action is (D1-002 §14.1) |
| Empty (`DASH-006`: no open positions) | Calm disclosure, never a "you should be trading" prompt (Constitution §3.2, `26` §3) |
| Error (provider/consumer failure) | `signal.warn` for degraded/partial, `signal.critical` only for fully blocking (D1-002 §14.4); factual copy with a timestamp for last-known data (D1-002 §14.6) |
| Success | As specified per block above |

# 8. Interaction Rules

Per D1-002 §12 (Decision Velocity): every interaction on Dashboard is
navigation-only — selecting a reading, a card, or a Peripheral label
moves the trader to another screen; none mutates state (`26` §4.3,
Interaction Map). All are therefore **low-stakes, reversible**
interactions (D1-002 §12.1) — fully frictionless, no confirmation step,
since D1-002 §12.2's higher-friction rule applies only to consequential,
evidence-weighing actions, none of which exist on Dashboard itself.

## 8.1 Visual weight vs. Attention/Action hierarchy

`DASH-002`'s conclusion text carries the strongest weight on the page
(largest `text.heading`, most generous surrounding whitespace,
`space.64` isolation) per Design Constitution rule 6 — assigned by
Attention level (`M4-003` §3), never by favorability, novelty, or how
recently a reading was computed (Constitution §8.3). No button on
Dashboard is styled Primary-tier (D1-002 §13) in the Action Hierarchy
sense, since no consequential action exists here — every affordance is
Tertiary-tier at most (D1-003 §3.1: text-only, neutral color, no accent
fill), consistent with Dashboard being a synthesis surface, not an
action surface (`M4-002` §8).

# 9. Motion Rules

Per D2-001 §7–§8 and D1-002 §5:

- A reading changing value (e.g. `DASH-002`'s own conclusion updating)
  uses `motion.duration.default` (200ms) cross-fade only — never a
  flash, shake, or color pulse, regardless of the magnitude of the
  underlying change (D1-002 §5.3).
- Skeleton-to-content transition uses `motion.duration.default`,
  `motion.easing.standard`.
- Every transition has a `prefers-reduced-motion` equivalent per D2-001
  §8 — instant (0ms) state change.
- No entrance animation plays on Dashboard load beyond the
  Loading→Success transition already specified — no decorative
  auto-playing motion (D1-004 §4 rule 4.3).

# 10. Accessibility Rules

Per D2-007, applied to Dashboard:

- One `<main>` landmark, one `<h1>` (`26` §3, `DASH-001` Accessibility).
- `DASH-002`'s synthesized statement is announced as one coherent
  sentence, not a disconnected grid (`26` §3, `DASH-002` Accessibility).
- Nested `DASH-003`'s confidence kind and value are announced as one
  semantic unit (D2-007 §6.2, mirroring Status Chip behavior).
- Every Card (`DASH-004`/`005`/`006`) and Peripheral label is keyboard-
  reachable, in an order matching the Attention Hierarchy reading order
  (D2-007 §2.2) — Primary, then the three Secondary cards in the order
  fixed at `M4-003.1` §2.4 (Portfolio, Watchlist, Morning Brief), then
  the Peripheral row.
- Focus indicator: `border.emphasis`, 2px, `accent.focus` on every
  focusable element, never suppressed (D2-007 §4).
- No information anywhere on Dashboard is conveyed by color alone
  (D2-007 §5) — gain/loss leads with `+`/`−` symbol and tabular figures
  (D2-003 §5.3), `signal.warn`/`signal.critical` conditions always pair
  with a text label (D2-005 §14 constraint).
