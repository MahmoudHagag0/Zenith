## ADR-001 — JWT-Based Authentication (S1-001 Foundation)

-   **Status:** Approved
-   **Context:** `documentation/zos/sprints/S1-001_SPRINT_BRIEF.md` Scope item 5 required a baseline authentication mechanism for `apps/api`, but no ADR previously established which mechanism was approved. `16_NAMING_CONVENTIONS.md` contained JWT-related naming examples suggesting, but not formally establishing, JWT.
-   **Decision:** S1-001's authentication foundation will use JWT-based authentication. Scope is limited to authentication foundation, authentication middleware/base services, and protected-route capability.
-   **Consequences:** `apps/api` implements JWT issuance/validation as its baseline authentication mechanism. No other authentication mechanism is authorized for S1-001.
-   **Alternatives Considered:** None recorded beyond the JWT naming precedent already present in `16_NAMING_CONVENTIONS.md`; no competing mechanism was proposed.
-   **Related Components:** `apps/api`.
-   **Related Decision Log Entry:** DEC-2026-001.

Explicitly out of scope under this ADR: OAuth providers, social login, advanced identity management, and any authentication complexity beyond this baseline. Introducing any of these requires a superseding ADR.

## ADR-002 — Argon2id Password Hashing (S1-002 User Management)

-   **Status:** Approved
-   **Context:** S1-002 introduces the first real `User` domain model and self-registration/login, requiring a password-hashing mechanism. No ZOS document previously specified one, and `04_TECH_STACK.md` does not list a hashing library.
-   **Decision:** User passwords are hashed with Argon2id. No other hashing algorithm (e.g. bcrypt, scrypt, plain SHA) is authorized for S1-002.
-   **Consequences:** `apps/api`'s registration/login flow depends on an Argon2id-capable library (subject to `14_DEPENDENCY_POLICY.md` review at implementation time). Password hashes are never logged or exposed via API responses.
-   **Alternatives Considered:** None recorded beyond the Architecture Team's direct selection of Argon2id; no competing algorithm was proposed for evaluation.
-   **Related Components:** `apps/api`, `packages/database` (`User.passwordHash`).
-   **Related Decision Log Entry:** DEC-2026-003.

## ADR-003 — Market Data Provider Abstraction (S1-005 Market Data Foundation)

-   **Status:** Approved
-   **Context:** S1-005 requires Zenith to source quotes and historical candles for catalog assets. `04_TECH_STACK.md` does not list any market-data vendor, and no external paid or free market-data service has been reviewed or approved by the Architecture Team. Constitution Rule 1 (no undocumented architecture) prohibits silently integrating an unapproved external dependency, and the Sprint's own instructions explicitly disallow external paid services unless already approved. At the same time, the product goal requires a real, working market-data foundation now, not a stalled sprint waiting on a future vendor-selection decision.
-   **Decision:** Market data is sourced through a `MarketDataProvider` interface (`getQuote`, `getCandles`, `checkHealth`), so that `apps/api`'s business logic (caching, rate limiting, retry, background sync, the HTTP API) depends only on this interface, never on a concrete vendor. The only implementation shipped in this sprint is a `SimulatedMarketDataProvider` — a deterministic, in-process generator of quotes and candles (seeded by asset symbol and time bucket, not random), clearly labeled in code and documentation as simulated, not real market data. No real external market-data vendor is integrated by this decision.
-   **Consequences:** Every quote and candle returned by S1-005's API is simulated, not a real market price, until a future ADR selects and integrates a real vendor behind this same interface — at which point only a new `MarketDataProvider` implementation and its module registration change; caching, rate limiting, retry, background sync, and the HTTP API are unaffected. This is a deliberate, transparent placeholder, not a claim of real market data.
-   **Alternatives Considered:** (1) Integrate a free, no-API-key public market-data endpoint directly — rejected because no such vendor has been reviewed or approved by the Architecture Team, and doing so without that review would be an undocumented architecture change and an unapproved external dependency. (2) Defer the entire sprint until a vendor is approved — rejected because the caching, retry, rate-limiting, background-sync, and API-surface foundation this sprint builds is valuable and testable independent of which provider eventually supplies real data, and building it now against a stable interface directly enables a future vendor swap with no business-logic change.
-   **Related Components:** `apps/api/src/market-data` (provider interface and `SimulatedMarketDataProvider`).
-   **Related Decision Log Entry:** DEC-2026-006.

Explicitly out of scope under this ADR: selecting or integrating any real external market-data vendor. Introducing one requires a superseding or additional ADR at the time a vendor is actually reviewed and approved.

## ADR-004 — Background Job Scheduling for Market Data Synchronization (S1-005)

-   **Status:** Approved
-   **Context:** S1-005 requires periodic background synchronization of market data for assets traders actually track (watchlisted, favourited, or held), so cached quotes do not silently go stale between requests. No prior sprint introduced any background/scheduled job execution capability, and `04_TECH_STACK.md` does not list a scheduling mechanism.
-   **Decision:** Background synchronization uses `@nestjs/schedule`, the official NestJS scheduling module, via its `@Cron()` decorator. No other scheduling mechanism (custom `setInterval` loops, an external job queue, a separate worker process) is authorized for this sprint.
-   **Consequences:** `apps/api` gains a `MarketDataSyncService` that periodically refreshes cached quotes for tracked assets only (not the entire catalog), respecting the same rate limiter and retry logic as on-demand requests. `@nestjs/schedule` becomes a new runtime dependency, reviewed under `14_DEPENDENCY_POLICY.md` at implementation time (official NestJS-maintained package, no license concern, minimal footprint, no overlapping existing dependency).
-   **Alternatives Considered:** An external job queue (e.g. BullMQ with Redis) was considered and rejected as disproportionate to this sprint's needs and would introduce a new infrastructure dependency (Redis) not present in `04_TECH_STACK.md`; a hand-rolled `setInterval` loop was considered and rejected because it duplicates functionality `@nestjs/schedule` already provides as a maintained, officially-supported NestJS module, contrary to `04_TECH_STACK.md`'s "prefer official libraries" rule.
-   **Related Components:** `apps/api/src/market-data` (`MarketDataSyncService`).
-   **Related Decision Log Entry:** DEC-2026-007.

## ADR-005 — Shared Deterministic Computation Infrastructure (Indicator Engine, Swing Detection, Regime/Context Service)

-   **Status:** Proposed
-   **Context:** The Analysis Engine (`documentation/zos/22_ANALYSIS_ENGINE_ARCHITECTURE.md`) requires computation shared across many future Analysis Providers. Architecture research established: (1) nearly every published technical indicator (RSI, MACD, ATR, ADX, Bollinger Bands, etc.) has exactly one canonical formula and must be computed once, not reimplemented per Provider — critically, J. Welles Wilder Jr.'s original 1978 RSI/ATR/ADX smoothing differs from the plain EMA/SMA approximation many third-party implementations silently substitute, and MACD's histogram is a distinct 1986 addition (Thomas Aspray) to Gerald Appel's original line, requiring separate source attribution; (2) swing/pivot detection is required by roughly 12 of the ~13 methodology families identified as future Providers (Market Structure, Support/Resistance, Elliott Wave, Fibonacci, Harmonic Patterns, Chart Patterns, and others), making it the single most universal dependency in the system, more so than any individual indicator; (3) several future Providers (Mean Reversion, Breakout Methodology, Wyckoff, VSA) each independently need a "current market regime" (trending vs. ranging, volatility state) read to gate their own confidence, and deriving this ad hoc per-Provider would recreate the exact duplicated-computation problem this ADR exists to prevent. The Regime/Context Service specifically originated as an Implementation-Engineer architecture-research finding, not from the Architecture Team's original revision requests for the Analysis Engine; it is bundled into this ADR for infrastructure-layer cohesion — all three components share an identical "pure computation, no interpretation, point-in-time-deterministic" character — but its provenance is recorded here distinctly from the Indicator Engine's so the audit trail is not misread as entirely Architecture-Team-originated.
-   **Decision:** Zenith introduces three shared, deterministic computation services, consumed by Analysis Providers exclusively via interface/token-based dependency injection (never concrete-class injection), following the `MARKET_DATA_PROVIDER` precedent of ADR-003: (1) **Indicator Engine** — SMA, EMA, RSI, MACD (line and histogram tracked as separately source-attributed outputs), Bollinger Bands, ATR, ADX, a Fibonacci ratio calculator, and a Donchian channel calculator; RSI/ATR/ADX use Wilder's original recursive smoothing exactly, and the engine is implemented as an internal registry of individual indicator calculators, never one monolithic service class. (2) **Swing Detection Infrastructure** — a single parameterized swing/pivot detector with disclosed, never-silently-defaulted sensitivity, producing Market Structure primitives (swing sequence, BOS/CHoCH) directly; must be point-in-time deterministic, confirming a swing only once enough subsequent bars exist under the fixed sensitivity rule, identically in live execution and backtesting. (3) **Regime/Context Service** — composes Indicator Engine (ADX, ATR) and Swing Detection (Market Structure) output into a single versioned trend/volatility regime read, consumed by Providers before they compute their own confidence; must also be point-in-time deterministic. All three consume market data exclusively through a `MarketSeries`/`PriceSeries` value object translated from `Candle`/`MarketQuote` at the Analysis Engine's boundary — an Anti-Corruption Layer owned by the Analysis Engine, never by the Market Data domain (full specification in `22_ANALYSIS_ENGINE_ARCHITECTURE.md`) — so that a future change to `Candle`/`MarketQuote` (e.g. the already-anticipated intraday data-model expansion) is absorbed by the translation adapter rather than propagating into every computation. All three additionally return mandatory computation metadata (parameters used, formula/source citation, input data range, computation timestamp, and intermediate values where applicable) and a `computationVersion`, versioned independently of any consuming Provider's `contractVersion`, so a future formula correction (e.g. a corrected smoothing constant) is recorded without requiring any output shape to change. All three are pure computation with no interpretation layer; interpretation remains the exclusive responsibility of Analysis Providers under ADR-006.
-   **Consequences:** `apps/api` gains three new shared services with zero Prisma schema changes (pure computation over the existing `Candle`/`MarketQuote` data already introduced by S1-005, reached only through the `MarketSeries` translation adapter). Every future Analysis Provider (S1-009 onward) depends on these services rather than reimplementing indicator, swing, or regime logic. A shared cache keyed by `(computation, parameters, instrument, data-range)` is required once more than a handful of Providers consume the same output, to avoid redundant recomputation at scale. Regime/Context Service threshold calibration (ADX trending cutoff, ATR volatility percentile) is deferred to a Decision Log entry at S1-007 implementation time, not fixed by this ADR. Point-in-time determinism for Swing Detection and Regime/Context Service is a hard requirement of this ADR and must be verified during S1-007 implementation before any Provider or future Backtesting capability depends on it. No new runtime dependency is introduced — consistent with `04_TECH_STACK.md` and `14_DEPENDENCY_POLICY.md`, this is pure TypeScript computation within the existing NestJS/Prisma stack. This revision is additive and backward-compatible with the originally drafted ADR: no previously stated decision is reversed, and the `MarketSeries` boundary and computation-metadata/`computationVersion` requirements apply from S1-007's first implementation rather than being retrofitted later.
-   **Alternatives Considered:** (1) Let each Analysis Provider compute its own indicators/swings/regime independently — rejected, this is precisely the duplicated-computation and semantic-drift problem (e.g. inconsistent RSI smoothing across Providers) this ADR exists to eliminate. (2) Defer Swing Detection Infrastructure to the Analysis Provider Framework (ADR-006) rather than this ADR, treating it as framework-adjacent rather than pure computation — rejected, because swing detection has no interpretation layer and shares the Indicator Engine's exact "pure computation, disclosed parameters, point-in-time-safe" character; bundling it here keeps ADR-006 focused purely on interpretation and plugin mechanics. (3) Give the Regime/Context Service its own separate ADR, given its different origin (Implementation-Engineer research finding rather than Architecture Team request) — rejected in favor of one infrastructure ADR with explicit internal provenance attribution (see Context), to avoid fragmenting three components that share identical architectural character and identical point-in-time-determinism obligations across multiple ADRs. (4) Let the Indicator Engine, Swing Detector, and Regime/Context Service consume `Candle`/`MarketQuote` directly, without a translation boundary — rejected per `05_ARCHITECTURE.md`'s existing dependency rules (domain logic must not depend directly on infrastructure/persistence entities of another bounded context); the `MarketSeries` Anti-Corruption Layer is the same principle ADR-003 already applies to external vendors, applied here to Zenith's own internal Market Data domain boundary.
-   **Related Components:** `apps/api/src/analysis-engine/indicator-engine`, `apps/api/src/analysis-engine/swing-detection`, `apps/api/src/analysis-engine/regime-context` (anticipated paths; final module structure is an S1-007 implementation decision, not fixed by this ADR).
-   **Related Decision Log Entry:** None yet recorded; anticipated at S1-007 implementation time (swing-detection sensitivity default, Wilder-smoothing conformance note, Regime/Context Service threshold calibration, `MarketSeries` field-list finalization, and the `computationVersion` numbering scheme — see `22_ANALYSIS_ENGINE_ARCHITECTURE.md`, Known Limitations and Additional Findings).

Explicitly out of scope under this ADR: any interpretation, Evidence, or confidence logic (see ADR-006); any normalized-vocabulary or Confluence-aggregation logic (see ADR-007); any real market-data vendor decision (already governed by ADR-003).

## ADR-006 — Analysis Provider Plugin Architecture & Standard Output Contract

-   **Status:** Proposed
-   **Context:** `documentation/zos/22_ANALYSIS_ENGINE_ARCHITECTURE.md`'s Analysis Provider Framework and Evidence/Interpretation/Limitations Contract sections require formal ADR backing before S1-008 (Analysis Provider Framework sprint) begins. Architecture research and the Final Architecture Review identified: Providers are not always mutually independent (VSA depends on Wyckoff's active range output), so a flat parallel plugin registry is insufficient; some methodologies (Elliott Wave, Wyckoff's Phase-schematic layer) are irreducibly multi-hypothesis, so a single-interpretation-per-Provider contract would misrepresent them; confidence is not one concept across methodologies (detection quality, per-hypothesis ranking, regime-gating, and source-quality ceiling are four distinct concerns that must not be collapsed into one comparable scalar); and every future consumer (Risk Engine, Decision Engine, AI Engine) will depend on this contract's shape, with no existing mechanism to introduce a breaking change safely.
-   **Decision:** Analysis Providers are registered as an additive NestJS multi-provider set, each implementing a common `AnalysisProvider` interface. Provider registration includes an optional declared-dependency list on other Providers' outputs, referenced **by stable Provider identifier/token, never by importing another Provider's concrete class** — the Execution Engine resolves tokens to instances at runtime, keeping no Provider compile-time coupled to another's implementation; the framework's Execution Engine performs a topological sort over these declarations before invocation, and tags each Provider fast-tier or slow-tier so an expensive Provider (e.g. Elliott Wave's multi-hypothesis search) never blocks fast-tier results. Provider registration additionally includes an optional self-declared `methodologyFamily` identifier, authored exclusively by the Provider and read (never assigned) by the Confluence Engine (ADR-007), and a `computationVersion` (distinct from `contractVersion`; see the Determinism/Backtesting rationale established in ADR-005) that increments whenever the Provider's own interpretation logic changes in a way that could alter previously computed results. A Provider that fails or times out repeatedly is placed into an open-circuit state and temporarily excluded from invocation rather than re-invoked on every request, reusing the retry/backoff philosophy ADR-003 already established for external calls, applied here to in-process Provider invocation; a Provider in an open-circuit state is reported as non-participating, never silently dropped. Every Provider returns a standard contract: `contractVersion` (mandatory, enabling non-breaking future evolution for downstream consumers); `evidence` (detected/missing conditions, supporting/conflicting evidence); `interpretation` (an **array**, always — length one for single-hypothesis Providers, length N for multi-hypothesis ones such as Elliott Wave or Wyckoff's Phase layer — preserving uniform substitutability for every Consumer rather than treating multi-hypothesis output as a special case); `limitations` (data-quality caveats, methodology confidence ceilings, stated assumptions); a `traceability` record (raw data references, intermediate calculations, condition derivations, confidence derivation); and confidence fields explicitly labeled as one of four kinds — Detection Confidence, Interpretation Confidence (per hypothesis), Regime-Adjusted Confidence, or Methodology Confidence Ceiling — never an unlabeled scalar. The `AnalysisProvider` interface additionally requires a `normalize()` method; **this ADR establishes only that the method exists as part of the interface — its target vocabulary, versioning, and conformance requirements are defined exclusively by ADR-007**, avoiding overlapping ownership between the two ADRs. A Provider facing missing or unusable data must return a populated `limitations` entry, never throw. Each Provider carries a lifecycle state — ACTIVE, DEPRECATED, or RETIRED — governing whether it participates in new live runs versus remaining available only for reproducing historical (including backtested) results. Where a Provider is multi-hypothesis, the maximum number of `interpretation` entries it may return is bounded and recorded via Decision Log at that Provider's own implementation time, not fixed by this ADR, to prevent unbounded combinatorial growth (e.g. Elliott Wave wave-count search).
-   **Consequences:** `apps/api` gains an Analysis Provider Framework module (registry, execution engine, base contract types) with zero Prisma schema changes (Provider output is computed, not persisted, consistent with the Analytics layer's S1-006 precedent). Every future Provider sprint (S1-009 onward) implements this exact contract; a Consumer (Dashboard, Alerts, Risk Engine, Decision Engine, AI Engine) can treat every Provider polymorphically because `interpretation` is always an array. Introducing a Provider that depends on another Provider's output (the VSA/Wyckoff relationship already anticipated) requires only a dependency declaration, not a special-cased execution path. Retiring a Provider requires a Decision Log entry recording rationale and last-active contract version, per the Provider Lifecycle rule. The Execution Engine must also expose per-Provider latency, failure/timeout rate, and circuit-breaker state as observable metrics (see `22_ANALYSIS_ENGINE_ARCHITECTURE.md`, "Operational Resilience & Observability") — a specification requirement, not an infrastructure/tooling choice. No new runtime dependency is introduced. This revision is additive and backward-compatible with the originally drafted ADR: the base contract shape (`contractVersion`, `evidence`, `interpretation[]`, `limitations`, `traceability`, the four-part confidence taxonomy) is unchanged; `computationVersion`, `methodologyFamily`, token-based dependency references, and the circuit breaker are new, independently-versioned additions to the same interface.
-   **Alternatives Considered:** (1) A flat, fully-independent Provider registry with no dependency declarations — rejected, since at least one already-anticipated Provider relationship (VSA → Wyckoff) violates the independence assumption; a naive parallel-only model would require an ad hoc workaround the first time it was implemented. (2) A single-interpretation-per-Provider contract with multi-hypothesis Providers treated as a special-cased optional extension — rejected in favor of always returning `interpretation` as an array, since an optional extension breaks Consumer polymorphism (every Consumer would need to special-case the multi-hypothesis Providers), whereas a uniform array-typed field costs nothing for single-hypothesis Providers and preserves substitutability for all. (3) A single unlabeled `confidence` scalar per Provider, deferring the four-part taxonomy to a later refinement — rejected, since displaying two differently-computed "70%" values as directly comparable was identified as a concrete trust-model risk during architecture review, and retrofitting the taxonomy after Consumers already depend on an unlabeled scalar would itself be a breaking change this ADR's own `contractVersion` mechanism is meant to make rare, not routine. (4) Reference Provider dependencies by concrete class import rather than a stable token — rejected, since it would compile-time couple the Execution Engine and every dependent Provider to specific Provider implementations, violating Dependency Inversion and undermining the Provider Lifecycle's retirement/replacement model. (5) Re-invoke a repeatedly-failing Provider unconditionally on every request rather than applying a circuit breaker — rejected as unsustainable at 50+ Providers and millions of analyses; a circuit breaker reuses an already-approved pattern (ADR-003's retry/backoff philosophy) rather than introducing a new one.
-   **Related Components:** `apps/api/src/analysis-engine/providers` (registry, execution engine, base `AnalysisProvider` interface and contract types — anticipated path, final structure is an S1-008 implementation decision).
-   **Related Decision Log Entry:** None yet recorded; anticipated at S1-008 implementation time (Provider dependency-declaration syntax, fast-tier/slow-tier classification criteria, initial confidence-label naming conventions, per-Provider multi-hypothesis bound, per-Provider `computationVersion` numbering scheme, and circuit-breaker failure-threshold/reset-timeout defaults). Whether a DEPRECATED Provider's `computationVersion` may still increment is an open question — see `22_ANALYSIS_ENGINE_ARCHITECTURE.md`, Additional Findings (Finding B).

Explicitly out of scope under this ADR: the shared indicator/swing/regime computations Providers consume (see ADR-005); the normalized vocabulary `normalize()` translates into, and Confluence aggregation logic (see ADR-007).

## ADR-007 — Confluence Architecture: Normalization & Weighting-Readiness

-   **Status:** Proposed
-   **Context:** `documentation/zos/22_ANALYSIS_ENGINE_ARCHITECTURE.md`'s Normalization and Confluence Engine sections require formal ADR backing before S1-012 (Confluence Engine sprint) begins, and before any earlier Provider sprint (S1-009–S1-011) implements a `normalize()` method — established by ADR-006 as part of the `AnalysisProvider` interface — against an unratified vocabulary. Architecture research established: different methodologies describe markets in incompatible vocabularies, but a centralized translation service would recreate the "must edit a central component for every new Provider" problem the plugin architecture (ADR-006) exists to avoid; different methodologies are not equally informative, so Confluence must not assume equal weighting is permanent, but implementing real differential weighting now is premature without historical validation data; and equal weighting alone creates a concrete false-confidence risk when near-duplicate methodologies (e.g. the merged ICT/SMC Provider alongside Supply/Demand-derived zone evidence) are counted as independent confirmations.
-   **Decision:** Each Analysis Provider's `normalize()` method (whose existence on the `AnalysisProvider` interface is established by ADR-006) translates its Evidence/Interpretation into a shared, centrally-specified vocabulary: Trend, Momentum, Liquidity, Structure, Volatility, Volume, Confirmation ("Risk" is explicitly excluded from this phase, reserved for a future Risk Engine). The vocabulary schema is versioned and additive: a new dimension may be introduced later via a Decision Log entry and schema version bump, without requiring any existing Provider's `normalize()` implementation to change (new dimensions default to "not applicable" for Providers that predate them). A shared conformance test suite — fixed input/output examples every `normalize()` implementation must satisfy — is maintained centrally alongside the vocabulary schema to prevent semantic drift across decentralized implementations. The Confluence Engine aggregates normalized signals via a pluggable `ConfluenceWeightStrategy` interface; only `EqualWeightStrategy` (weight 1.0 for every Provider) is implemented in this phase, so a future weighting strategy can be introduced without any Provider contract change. **Every `ConfluenceWeightStrategy` implementation, present or future, must return a `weightExplanation` alongside each generated weight**, so Confluence's explainability guarantee holds even once real differential weighting exists, not only for today's trivial equal-weight case. Providers are grouped into methodology families via a `methodologyFamily` field **self-declared by each Provider at registration (ADR-006) and only read by the Confluence Engine, which never assigns or edits family membership** — so that agreement within a family (e.g. the merged ICT/SMC Provider and Supply/Demand-tagged zone evidence) is never counted as independent confirmation, regardless of the active weighting strategy, and so that adding a Provider to an existing family never requires a Confluence Engine code change. Confluence output reports Provider participation explicitly (which Providers contributed vs. were unavailable) and includes a structured disagreement-explanation field computed **at the normalized-dimension level (Trend, Momentum, Liquidity, Structure, Volatility, Volume, Confirmation), never as pairwise Provider-vs-Provider comparison** — keeping this computation O(Providers × 7) rather than O(Providers²) as Provider count grows toward 50+ — distinguishing expected disagreement (e.g. trend-following vs. mean-reversion evidence) from unexpected inconsistency. Full per-Provider traceability is referenced by ID in Confluence output, not embedded, to bound response payload size as the Provider count grows toward 50+; the referenced trace record's retention/TTL/lifecycle model is specified in `22_ANALYSIS_ENGINE_ARCHITECTURE.md`'s Traceability section, not by this ADR.
-   **Consequences:** `apps/api` gains a Confluence Engine module with zero Prisma schema changes (aggregation is computed, not persisted). Every Provider sprint from S1-009 onward implements `normalize()` against this ADR's vocabulary from the start, rather than retrofitting it once S1-012 begins. Real differential weighting remains a future, separately-decided capability; this ADR only guarantees the contract will not need to change when that happens. Methodology-family grouping must be maintained as new Providers are added, per the Extension Guidelines already stated in `22_ANALYSIS_ENGINE_ARCHITECTURE.md`. Provider-participation rate, per-dimension disagreement frequency, and aggregation latency must be observable (see `22_ANALYSIS_ENGINE_ARCHITECTURE.md`, "Operational Resilience & Observability"). No new runtime dependency is introduced. This revision is additive and backward-compatible with the originally drafted ADR: `EqualWeightStrategy`'s behavior is unchanged (weight remains 1.0 for every Provider); `weightExplanation`, Provider-declared `methodologyFamily`, and dimension-level disagreement computation are clarifications and extensions of the same design, not a reversal of it.
-   **Alternatives Considered:** (1) A centralized normalization/translation service that all Providers are routed through — rejected, since it reproduces the exact "central component must be edited for every new Provider" anti-pattern ADR-006's plugin architecture is designed to avoid. (2) Implement real differential Confluence weighting immediately, using subjective methodology-quality judgments — rejected as premature without historical/backtested validation data, and inconsistent with this architecture's Determinism principle if the weights themselves are not empirically grounded. (3) Omit methodology-family grouping until real weighting exists — rejected, since this was identified as a live, day-one trust risk (equal-weighted agreement-counting over correlated Providers), not a problem only relevant once weighting is implemented. (4) Let the Confluence Engine own and hardcode methodology-family membership rather than reading a Provider-self-declared field — rejected, since it would require editing the Confluence Engine every time a new Provider joins an existing family or a new family is introduced, violating the Open/Closed principle this architecture otherwise maintains throughout. (5) Compute disagreement explanation as pairwise Provider-vs-Provider comparison rather than at the normalized-dimension level — rejected as an unnecessary O(Providers²) cost at 50+ Providers, with no additional explanatory value over dimension-level disagreement for how traders actually reason about conflicting evidence.
-   **Related Components:** `apps/api/src/analysis-engine/confluence` (aggregation engine, weight-strategy interface, normalized-vocabulary schema — anticipated path, final structure is an S1-012 implementation decision).
-   **Related Decision Log Entry:** None yet recorded; anticipated at S1-012 implementation time (initial methodology-family groupings, vocabulary schema version 1 definition, `EqualWeightStrategy` as the interim default, `weightExplanation` format conventions, and Trace Store retention/TTL defaults). Whether per-dimension disagreement explanations should list representative contributing Providers is an open question — see `22_ANALYSIS_ENGINE_ARCHITECTURE.md`, Additional Findings (Finding C).

Explicitly out of scope under this ADR: the `AnalysisProvider` interface itself and its Evidence/Interpretation/Limitations/Traceability/Confidence fields (see ADR-006); the shared indicator/swing/regime computations (see ADR-005); any real differential weighting strategy (requires a superseding or additional ADR once historical validation data exists).
